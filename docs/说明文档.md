# AI 动态追踪说明文档

## 1. 我想做的是什么，为什么这样定义“AI 动态追踪”
### 1.1 项目定位

这个网页重点强化对海外硅谷一手动态的快反捕捉，并把公开信息转换成可解释的产品判断。

这里的“AI 动态追踪”我主要定义为：

- 新产品发布
- 功能 / 版本更新
- 模型 / API / SDK 变化
- Agent 工作流升级
- 企业接入 / 平台开放
- docs / changelog / GitHub release 中体现能力演进的更新

这样定义的原因是：

1. 更贴近真实商业竞争  
真正反映竞争变化的，往往不是媒体评论，而是产品动作本身。

2. 更适合捕捉海外硅谷一手动态  
很多海外硅谷信号最先出现在官方博客、docs、changelog、GitHub release 和开发者社区，而不是传统媒体。

3. 更适合做趋势判断  
如果只追媒体热度，最后得到的是舆论摘要；如果追产品动作，就更容易看出能力迭代、产品形态和竞争焦点的变化。

## 2. 成品功能

这个成品的核心功能包括：

### 2.1 多源追踪

系统会持续追踪：

- 官方博客
- 官方 docs / changelog
- GitHub release
- 开发者社区信号
- 国内头部厂商官方产品页

并通过 `source_tier` 区分来源优先级：

- `T0`：官方博客 / docs / changelog / GitHub release
- `T1`：官方账号 / 创始人 / 核心负责人
- `T2`：社区首发
- `T3`：科技媒体
- `T4`：二次搬运

### 2.2 实时追踪能力

为了体现持续追踪能力，页面和系统层面都加入了实时机制：

- 最近成功更新时间
- 60 秒自动轮询
- 手动刷新
- 实时快讯流
- 来源池与来源健康状态
- 一手源占比 / 海外事件占比 / 高优先级事件数

### 2.3 可解释筛选与归并

系统不是简单聚合，而是加入了结构化判断：

- `novelty_score`
- `is_product_signal`
- `duplicate / same_event / event_update`

也就是说，它会判断：

- 这条内容是不是产品动态
- 这条内容是不是新信息
- 这条内容是不是和已有事件属于同一件事

### 2.4 趋势判断输出

系统会基于追踪结果生成：

- 趋势判断
- 证据快照
- 观察结论卡
- 实时日报
- 趋势周报

核心分析不只是摘要，而是基于 `trend_snapshot` 输出“核心变化、证据最强趋势、证据偏弱趋势”等内容。

## 3. 我是如何一步步完成这个成品的，主要利用了哪些 AI 工具

### 3.1 我采用的是多 Agent 开发思路

这个项目被拆成了 7 个 Agent：
1. 信息采集 Agent
2. 信号增强 Agent
3. 内容筛选 Agent
4. 事件归并 Agent
5. 趋势分析 Agent
6. 展示生成 Agent
7. 调度主控 Agent
完整链路是：

```text
collect -> enrich -> filter -> merge -> analyze -> present
```
#### 1. 信息采集 Agent
作用：

- 负责从公开源抓数据
- 优先抓海外硅谷一手源
- 支持增量抓取

对应你给 Cursor 的任务通常是：

- 读取 `sources.json`
- 支持官方博客、docs、changelog、GitHub releases、社区源
- 写入 `raw_items`

它解决的问题是：

- “有没有抓到新的信息”
#### 2. 信号增强 Agent
作用：

- 把原始内容变成结构化信号
- 提取 `company`、`product`、`event_type`、`keywords`
- 计算 `novelty_score`
- 标记 `evidence_type`

它解决的问题是：

- “这条信息到底是什么”
#### 3. 内容筛选 Agent
作用：

- 判断是不是 AI 产品动态
- 判断是不是值得分析的高价值信号

关键字段：

- `is_product_signal`
- `signal_score`
- `usefulness_score`
- `keep_for_analysis`

它解决的问题是：

- “该不该留下来”
#### 4. 事件归并 Agent
作用：

- 把同一事件的多源内容归并起来
- 把同一能力演进链中的补充更新识别出来

三种结果：

- `duplicate`
- `same_event`
- `event_update`

它解决的问题是：

- “这些内容是不是一回事”
#### 5. 趋势分析 Agent
作用：

- 基于归并后事件生成 `trend_snapshot`
- 输出趋势判断和结论卡

它解决的问题是：

- “最近 AI 产品到底在往哪里演进”
#### 6. 展示生成 Agent
作用：

- 把结构化结果变成页面和报告
- 输出 `dashboard_data.json`
- 输出日报 / 周报 / 观察结论卡

它解决的问题是：

- “怎么把结果讲清楚”
#### 7. 调度主控 Agent
作用：

- 统一调度前面 6 个 Agent
- 处理频率、顺序、失败恢复

完整顺序：

```text
collect
-> enrich
-> filter
-> merge
-> analyze
-> present
```

它解决的问题是：

- “系统怎么持续自动跑起来”

### 3.2 代码框架

我的做法：

1. 先用 AI 工具做架构规划  
先明确技术栈、目录结构、Agent 职责、数据表设计和 API 设计。

2. 再用 AI 工具逐步实现后端主链路  
先跑通抓取、增强、筛选、归并、趋势分析和展示生成。

3. 再用 AI 工具做静态主展示页  
优先做稳定、易部署、易答辩的静态展示页。

4. 再迭代增强实时感和趋势判断  
补最近成功更新时间、自动轮询、手动刷新、实时快讯流、趋势判断、日报周报等。

5. 最后补交付层  
包括运行脚本、部署脚本、GitHub Pages workflow、说明文档和展示材料。

### 3.3 主要利用的 AI 工具

我主要使用了以下工具：

- Cursor  
  用于项目结构搭建、分阶段编码、页面优化和多轮迭代。

- ChatGPT / Codex  
  用于方案细化、Agent 拆分、提示词设计、逻辑梳理、说明文档和答辩材料整理。

- GitHub / GitHub Actions  
  用于静态部署与定时刷新数据。

## 4. 附录

### 4.1 核心设计
#### 4.1.1 源头优先

系统对 source 做硬编码分层：

- `T0`：官方博客 / docs / changelog / GitHub release
- `T1`：官方 X / 创始人 / 核心负责人
- `T2`：Hacker News / Product Hunt / Reddit / 开发者社区
- `T3`：科技媒体
- `T4`：自媒体 / 二次搬运

`source_tier` 直接参与：

- 信号打分
- 主源选择
- 前端可信感展示
- 趋势分析证据强度

#### 4.1.2 双层时间机制

为了同时兼顾实时感与归并准确性，系统区分三类窗口：

- 实时信号窗口：3h / 6h / 12h
- 事件归并窗口：6h 到 36h
- 趋势分析窗口：7d / 30d / 90d

这样不会为了避免刷屏而牺牲前端的实时感。

#### 4.1.3 可解释筛选

系统不是直接按热度排序，而是先判断：

1. `is_product_signal`
2. 再计算 `signal_score / usefulness_score`

关键增强字段：

- `novelty_score`
  - 近 7 天同公司同产品语义相似度反向分
  - 新关键词 / 新能力描述
  - 更高 `source_tier` 的新证据
- `evidence_type`
  - `official_release`
  - `changelog`
  - `docs_update`
  - `github_release`
  - `founder_post`
  - `media_report`
  - `community_signal`

#### 4.1.4 三分法归并

事件归并不只是“合并/不合并”，而是：

- `duplicate`
- `same_event`
- `event_update`

`event_update` 表示同一能力演进链路中的补充更新，它不是重复，也不算完全独立的新主事件。

#### 4.1.5 趋势分析

趋势 Agent 先做 `trend_snapshot`，再写趋势判断。

快照固定包含：

- 各主题事件数
- 7d / 30d / 90d 环比
- 海外 / 国内占比
- 一手源占比
- 高价值事件中官方源占比
- 热门公司 Top N
- 热门产品 Top N
- 发布类 vs 更新类占比

最终输出 5 段：

- 核心变化
- 证据事件
- 背后原因
- 趋势判断
- 后续观察点
