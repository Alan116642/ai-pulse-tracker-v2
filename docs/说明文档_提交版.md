# 我具体是怎么做的？

这份说明文档我不想把过程写成“用了哪些框架、调了哪些参数”那种技术清单，而是尽量把我自己做这道题时真实的思考顺序还原出来。对我来说，这个题目最难的地方不是写一个页面，而是先判断 **到底该追什么、页面要帮助我看出什么、最后怎么把结果做成别人点开就能理解的东西**。

> **总的来说，我主要用了这几类 AI 工具（按使用强度排序）：**  
> **1. Cursor**：主力编码、调结构、改页面文案  
> **2. ChatGPT / Codex**：拆问题、排错、收敛提交材料  
> **3. DeepSeek**：辅助我检查中文表达和观点是否顺畅

## 在开始写代码前，我先想清楚三件事

### 1. 我到底要追什么？

一开始我没有把“AI 动态追踪”理解成普通新闻聚合。因为如果只是抓很多 AI 新闻，最后很容易做成一个看起来信息很多、但看完依然不知道重点的页面。所以我先把题目重新定义成 **AI 产品动作追踪**：重点看官方博客、产品页更新、GitHub release、开发工具更新，以及国内外头部平台的能力变化。

我做这个取舍的原因很现实：商业分析更关心产品变化和竞争方向，而不是资讯数量。媒体报道当然有价值，但如果我要判断“最近行业到底在往哪里走”，最值得优先看的，还是官方发布、产品页变化、API 更新和开发文档。

### 2. 页面到底要帮我回答什么问题？

我希望页面打开之后，能非常快地回答下面几个问题：

- 今天最值得看的 AI 产品动态是什么？
- 这些变化主要来自海外还是国内？
- 这些动作只是零散更新，还是已经形成一条趋势？
- 如果我要把今天和这周的变化讲给别人听，能不能直接沉淀成日报和周报？

所以我从一开始就不是按“新闻列表页”的思路做，而是按“重点 -> 判断 -> 证据 -> 输出”的顺序去设计页面结构。这样页面才能真正支持分析，而不是只负责展示。

### 3. 怎么降低别人体验这个产品的门槛？

我一开始就知道，如果别人还要先装环境、再跑命令，体验会很差。所以我最后故意做成三层交付：

- **在线链接**：一点开就能看到成品
- **本地 live 版**：如果需要验证完整抓取能力，再本地启动
- **GIF / 视频演示**：不想自己操作时，也能快速知道怎么用

这背后的思路其实很简单：平时只想看效果的人，点链接就够了；如果想验证“这个系统是不是真的在跑”，再用本地版就行。

## Step 1：先做最小可行性探路，而不是一上来写大页面

**动作：** 先快速验证哪些源能抓、抓回来的格式差异大不大、页面最小能不能显示出来。  
**目的：** 先确认这条路走得通，再决定后面要不要往上叠判断逻辑和更复杂的展示。

一开始我没有先写完整后端，而是先用 AI 工具帮我快速做了几轮“小探针”：拿 Google DeepMind、Anthropic、OpenAI、腾讯混元、阿里云百炼这些源做样本，看 RSS、博客页、产品页和 changelog 的结构差异到底有多大。这个阶段最关键的收获，不是代码本身，而是让我意识到：**不同站点的数据结构很不一样，如果不先把输入摸清楚，后面很容易在抓取阶段反复返工。**

所以这一阶段的重点，其实不是“多写代码”，而是“用最小代价排除错误方向”。

## Step 2：先把数据链路跑通，再慢慢加判断能力

**动作：** 先做到“抓到内容 -> 存下来 -> 页面能显示”，然后再加筛选、归并和趋势判断。  
**目的：** 避免一开始堆太多复杂逻辑，最后发现页面连稳定数据都没有。

我当时给自己定的最小链路是：

`公开源抓取 -> 原始内容落库 -> 基础字段抽取 -> 页面展示`

只有这一步稳定了，我才继续往上加：
- 这条内容是不是 AI 产品动态
- 是否值得保留
- 是否属于同一事件
- 能不能进入趋势判断

这也是为什么这个项目最后没有做成一个“很多规则拼在一起的大脚本”，而是拆成了信息采集、信号增强、内容筛选、事件归并、趋势分析、展示生成和调度主控 7 个 Agent。这样每一步都很清楚：前一步输出什么，后一步基于什么判断。

## Step 3：提示词不是为了“让模型自由发挥”，而是为了让它只做单一判断

**动作：** 在需要模型参与的地方，把 prompt 设计成“只完成一个明确动作”。  
**目的：** 降低输出发散和格式漂移，让模型更像一个稳定的判断助手，而不是自由写作者。

这是我在这次任务里很有感触的一点。以前很容易给大模型一个大而泛的提示词，让它“帮我总结一下这条新闻”，但这种写法经常会带来两个问题：一是输出太像公关文案，二是不同轮次结果不稳定。后来我把提示词拆成更小的动作，效果反而更稳。

我在这个项目里比较核心的几类提示词包括：

### 1. 产品相关性判断

我不会问模型“这条新闻重要吗”，而是先问：

> 这条内容是否与 AI 产品动作直接相关？请只回答“是”或“否”。

这样做的好处是，模型先做一个最基础的门槛判断，不会一上来就输出大段主观描述。这个判断对应到系统里，其实就是 `is_product_signal` 这一步。

### 2. 事件类型判断

当我确认它是产品动态之后，再让模型判断它属于哪种动作，比如：
- 新产品发布
- 功能更新
- 模型升级
- API / SDK 更新
- Agent 工作流升级

这一步的 prompt 我会故意写得很窄，强调“只从预设类型中选最接近的一项”。这样做的目的是保证后面的趋势统计可比，而不是每次被模型发明新分类。

### 3. 关键词 / 标签提取

这一步我也不会让模型随便写关键词，而是更强调：
- 提取 3 到 5 个最能代表产品动作的词
- 如果内容不够明确，就宁可少提，也不要泛泛地贴“大模型”“AI 创新”这种无效标签

这样后面页面上的主题统计和证据快照才会更有分析价值。

### 4. 趋势判断提示词

到趋势分析阶段，我也不会直接让模型“总结一下最近变化”，而是先给它结构化输入，再限定输出结构。比如我会要求它按照：
- 核心变化
- 证据事件
- 背后原因
- 趋势判断
- 后续观察点

这样的格式输出。这样模型写出来的内容更像分析结论，而不是泛摘要。

这部分我最想表达的是：**提示词工程在这个项目里，不是为了炫技，而是为了让模型每次只做清楚、可控的一步。**

## Step 4：页面不是做“好看”就够了，而是要能支持判断

**动作：** 把页面拆成重点层、判断层、证据层、输出层。  
**目的：** 让页面不只是告诉我“抓到了很多资讯”，而是帮助我更快形成判断。

页面结构最后被我拆成了几层：

- 顶部状态栏：先证明系统还在持续更新
- LIVE DESK：先看当天最值得看的产品动作
- 指标卡：快速判断海外 / 国内、官方源 / 社区源的分布
- 趋势判断 + 证据快照：先给结论，再给支撑
- 实时快讯流 + 重点来源池：把结论拉回事实层
- 日报 / 周报：把当天和近几周变化沉淀成更容易复述的输出

这个结构是我刻意设计的，因为我不想让页面变成“资讯越多越好”的形式。我更希望它像一个分析台：先抓重点，再形成判断，再验证判断，最后沉淀输出。这样页面本身就带有商业分析的节奏。

## Step 5：给系统留好兜底，不让一个源挂掉就把页面拖空

**动作：** 给抓取、增强和展示都留 fallback。  
**目的：** 让体验尽量稳定，不因为某个源 403 或某次增强失败就整页变空。

这一步其实很像真实项目。因为我测试时很快发现：
- 某些官方源会临时 403
- 某些页面结构会变
- 某些增强结果偶尔会不稳定

所以我最后补了几层兜底：
- 抓取失败时保留最近一次成功快照
- 模型增强不可用时回退到规则摘要和本地判断
- 静态页即使不连本地后端，也能展示最近一次发布的数据

这些设计不一定最“炫”，但很实用。因为一个追踪产品如果总是因为一个接口波动就整页空掉，别人第一次打开就会失去信任。

## Step 6：把“能跑”变成“能交、能看、能复述”

**动作：** 最后一轮不只改代码，还补齐提交体验。  
**目的：** 让别人打开时不用猜怎么用，也不用先理解一堆技术背景。

我最后做的几件事包括：
- 放在线链接，确保一点就能体验
- 做 GIF 和视频，让别人不操作也能知道页面怎么用
- 把简要分析和说明文档分开写，避免材料角色混在一起
- 把文档语言收得更像人的复盘，而不是技术说明书

这一步对我来说很重要，因为这道题最终考察的不是“我会不会写出一段代码”，而是“我能不能把想法真正交付成一个别人能看懂、能体验、能复述的结果”。

## 这个项目最能体现我什么能力

如果只选一个点，我觉得这个项目最能体现的，不是“我会不会用某个框架”，而是 **我能不能把一个很泛的问题，拆成一条能落地、能交付、能解释清楚的路径**。

这道题一开始其实很模糊：AI 动态追踪到底做成新闻页、监控页，还是分析工具？我最后能把它收成现在这版，靠的不是某一个很炫的技术点，而是一连串取舍：

- 我先定义“追 AI 产品动作”，而不是追所有 AI 新闻
- 我先追一手源和产品页，而不是先抓一堆媒体
- 我先把数据链路跑通，再去堆页面效果
- 我把页面做成“先看重点，再看判断，再看证据，最后看输出”
- 我把在线链接、live 版、GIF 和视频都补齐，尽量降低别人体验门槛

如果一定要说 AI 工具在这里帮了我什么，我会说：**AI 工具帮我加快了执行速度，但没有替我做关键选择。** 真正决定项目方向和质量的，还是我怎么拆问题、怎么做取舍、怎么定义“这个页面到底算不算有用”。

## 最后一句话

如果用一句最接近我真实想法的话来总结这个项目，我会说：

**我做的不是一个把 AI 新闻堆在一起的页面，而是一个尽量把“公开信息 -> 结构化信号 -> 趋势判断 -> 可读输出”这条链路跑通的小工具。**  
它未必是一个完全成熟的产品，但比较完整地体现了我如何定义问题、如何拆路径、如何用 AI 工具辅助实现，以及如何把结果做成别人能直接看懂和体验的形式。
